# 计算机网络实验报告

## 第一部分：流量统计软件（流量地图）

本部分对应课程第三次实验“协议分析程序的编写”。实验要求基于 WinPcap/Npcap 抓取网络数据包，根据 IP 协议解析每个数据包的 PCI，并对不同 IP 地址进行流量统计。本项目在完成基本要求的基础上，进一步实现了基于 ECharts 的流量地图展示。

### 1. 实验的目的

本实验围绕课程中“理解协议在通信中的作用、掌握抓包软件开发、掌握协议解析编程方法”的目标展开。项目基于 Npcap/Scapy 实现数据包捕获，基于 IPv4 首部格式完成协议控制信息 PCI 解析，并进一步统计不同 IP 地址的收发包数量和字节数，最终通过 Web 页面和流量地图展示分析结果。

通过本实验，需要达到以下目标：

1. 理解网卡抓包程序的基本工作流程，包括查找网卡、选择网卡、打开网卡、设置过滤规则、循环读取数据包和显示结果。
2. 掌握 Ethernet II 帧和 IPv4 数据报的基本结构，能够从原始字节流中解析协议控制信息 PCI。
3. 能够按 IP 地址统计发送包数量、接收包数量、发送字节数、接收字节数等流量指标。
4. 能够按源 IP、目的 IP 和协议类型聚合通信流，形成 Flow 统计结果。
5. 理解公网 IP、私网 IP、组播地址和广播地址在地理定位上的差异。
6. 能够将 IP 到 IP 的通信关系转换为地点到地点的流量关系，并使用地图进行展示。
7. 理解静态抓包演示和实时抓包展示的区别，能够根据不同演示场景选择合适的数据来源。

本实验不仅完成了协议分析程序，还进一步将实验任务扩展为前后端流量地图系统，使网络协议分析结果能够以更直观的方式展示。

### 2. 流量统计软件（流量地图）的功能需求描述

根据实验要求，流量统计软件至少需要完成两个核心任务：第一，能够显示每个数据包的 PCI；第二，能够显示每个 IP 地址的流量，特别是接收包数量。在此基础上，本项目进一步实现了流量地图，将不同地点之间的数据发送量可视化。

#### 2.1 抓包功能

本项目实际采用 Npcap 作为抓包驱动环境。Npcap 与 WinPcap 在抓包编程思想上基本一致，均用于为应用程序提供原始数据包捕获能力，本实验使用 Npcap 完成课程中 WinPcap/Npcap 编程部分的要求。抓包功能包括：

1. 查找本机可用网卡设备列表。
2. 选择需要抓包的网卡，例如 WLAN 无线网卡。
3. 打开网卡并启动抓包。
4. 设置 BPF 过滤规则，例如 `ip`，只抓取 IP 数据包。
5. 循环读取数据包。
6. 将抓到的数据包保存为 `.pcap` 文件，便于后续离线分析。

#### 2.2 PCI 解析功能

软件需要对抓到的 `.pcap` 文件进行协议解析。解析内容包括：

1. Ethernet II 帧的目的 MAC、源 MAC 和 EtherType。
2. IPv4 首部中的版本号、首部长度、服务类型、总长度、标识、标志位、片偏移、TTL、协议号、首部校验和、源 IP、目的 IP。
3. 对 TCP 和 UDP 数据包继续解析源端口和目的端口。

解析结果写入：

```text
实验三_IP协议分析与流量统计/outputs/packet_pci.csv
```

4. 接收字节数。

#### 2.4 Flow 流量统计功能

IP 统计只能反映单个地址的收发总量，而 Flow 统计可以保留通信方向，因此后续可以将“源 IP → 目的 IP”的通信关系映射为“源地点 → 目的地点”的地图连线。

```text
源 IP + 目的 IP + 协议类型
```

每条 Flow 记录包括：

1. 源 IP。
2. 目的 IP。
3. 协议类型。
4. 包数量。
5. 字节数。

Flow 统计结果写入：

#### 3.5 协议解析模块设计

协议解析模块首先读取 PCAP 文件的全局头，识别字节序和链路层类型；随后逐个读取数据包记录。对于 Ethernet II 帧，程序通过 EtherType 判断是否为 IPv4。如果是 IPv4，则继续解析 IPv4 首部。

IPv4 首部解析重点包括：

1. Version：IP 版本号，本实验主要解析 IPv4。
2. IHL：首部长度，用于计算 IP 首部实际字节数。
3. Total Length：IP 数据报总长度。
4. Identification：分片标识。
5. Flags 和 Fragment Offset：分片控制字段。
6. TTL：生存时间，用于防止数据包在网络中无限循环。
7. Protocol：上层协议，如 TCP、UDP、ICMP。
8. Header Checksum：首部校验和。
9. Source IP 和 Destination IP：源 IP 与目的 IP。

如果 Protocol 表示 TCP 或 UDP，程序继续读取源端口和目的端口。

图12 IPv4 数据报首部结构及本系统解析字段

图中标出：

```
Version, IHL, TOS, Total Length, Identification, Flags, Fragment Offset, TTL, Protocol, Header Checksum, Source IP, Destination IP
```

本系统按照 IPv4 首部字段顺序从原始字节流中提取协议控制信息，并将其作为 PCI 结果输出到表格中。

#### 3.6 流量统计模块设计

流量统计模块基于解析出的 IPv4 包进行统计。每解析到一个 IPv4 包，系统会执行以下更新：

```text
源 IP 的发送包数量 +1
源 IP 的发送字节数 + 当前 IP 包长度
目的 IP 的接收包数量 +1
目的 IP 的接收字节数 + 当前 IP 包长度
源 IP -> 目的 IP 的 Flow 包数量 +1
源 IP -> 目的 IP 的 Flow 字节数 + 当前 IP 包长度
```

其中，“目的 IP 的接收包数量 +1”对应实验要求中“显示每个 IP 地址接收包数量”的核心指标。

#### 3.7 地理位置与地图模块设计

地理位置模块将 IP 地址映射为地点和经纬度。地图模块再将 IP 维度的 Flow 转换为地点维度的 Flow。

转换过程如下：

```text
源 IP -> 目的 IP
转换为
源地点 -> 目的地点
```

例如：

```text
例如，在本地映射表 `geo_overrides.csv` 中，可以将本机私网地址或校园网出口地址手动映射为“武汉”；公网目的 IP 则可以通过本地映射表或在线接口解析为对应城市。经过映射后，通信关系可表示为：
武汉 -> 贵阳
```

地图展示时，地点作为节点，地点间 Flow 作为带方向的连线。连线越粗或动态效果越明显，表示该方向字节数越多。

## 第二部分：蓝牙定位（测距）系统的设计方案

### 1. 设计目标

蓝牙定位（测距）系统的目标是在室内环境中利用蓝牙无线通信信号估计目标设备与多个参考节点之间的距离，并进一步计算目标设备的位置。该系统可以用于实验室设备定位、人员定位、资产管理、室内导航等场景。

本设计方案需要结合计算机网络知识，从协议分层、地址标识、无线信道、数据封装、通信流程、误差控制和系统安全等角度进行设计。

系统设计目标包括：

1. 能够识别目标蓝牙设备。
2. 能够采集目标设备的无线信号强度 RSSI。
3. 能够根据 RSSI 或蓝牙测距机制估计距离。
4. 能够通过多个参考节点进行位置计算。
5. 能够将定位结果通过网络传输到服务器。
6. 能够在前端页面显示目标位置。
7. 能够考虑无线环境中的干扰、丢包、时延和安全问题。

### 2. 蓝牙定位的网络基础

蓝牙是一种短距离无线通信技术，主要工作在 2.4GHz ISM 频段。与 Wi-Fi、以太网一样，蓝牙通信也可以从网络分层角度理解。

从计算机网络角度看，蓝牙定位系统涉及以下层次：

| 网络层次 | 在蓝牙定位系统中的体现 |
|---|---|
| 物理层 | 2.4GHz 无线信号、发射功率、接收功率、信道干扰 |
| 数据链路层 | 蓝牙设备地址、广播帧、连接管理、差错检测 |
| 网络/传输层 | 网关将定位数据通过 IP、TCP/UDP、HTTP/MQTT 上传 |
| 应用层 | 设备识别、距离计算、位置计算、地图展示 |

蓝牙定位本身主要依赖物理层信号特征和链路层设备标识，而定位结果的上传、存储和展示则依赖 TCP/IP 网络。

### 3. 系统总体结构

系统由四类角色组成：

1. 待定位蓝牙设备：如手机、蓝牙标签、手环或 BLE Beacon。
2. 蓝牙参考节点：部署在固定位置的接收器，用于扫描目标设备信号。
3. 定位服务器：接收参考节点上传的数据，进行距离估计和位置计算。
4. 前端展示端：显示设备位置、距离估计和历史轨迹。

总体数据流程如下：

```text

-> 多个参考节点扫描广播包并记录 RSSI
-> 参考节点通过 IP 网络上传观测数据
-> 服务器清洗、滤波、距离估计
-> 服务器使用三边定位或指纹匹配计算位置
-> 前端地图显示目标位置
```

### 4. 硬件组成设计

#### 4.1 待定位设备

待定位设备可以采用 BLE Beacon 或支持蓝牙广播的手机。设备周期性发送 BLE 广播包，广播包中包含设备标识信息。

设备需要具备：

1. 唯一设备标识，如 MAC 地址、UUID、Major、Minor。
2. 稳定的广播间隔。
3. 可配置的发射功率。
4. 足够的电池续航能力。

为了保护隐私，实际系统中不应直接长期使用真实 MAC 地址作为用户身份，而应使用应用层分配的临时 ID。

#### 4.2 蓝牙参考节点

参考节点部署在室内固定位置，例如房间四角、走廊节点、门口位置。每个参考节点需要知道自己的坐标，例如：

```text
Anchor A: (0, 0)
Anchor B: (8, 0)
Anchor C: (0, 6)
Anchor D: (8, 6)
```

参考节点可由以下设备实现：

1. Raspberry Pi + 蓝牙模块。
2. ESP32。
3. 带蓝牙功能的电脑。
4. 专用 BLE 网关。

参考节点负责扫描蓝牙广播包，并记录：

1. 设备 ID。
2. RSSI。
3. 参考节点 ID。
4. 参考节点坐标。
5. 时间戳。
6. 广播包内容。

#### 4.3 定位服务器

服务器负责接收多个参考节点上传的数据，并进行计算。服务器模块包括：

1. 数据接收模块。
2. 设备管理模块。
3. RSSI 滤波模块。
4. 距离估计模块。
5. 位置计算模块。
6. 数据库存储模块。
7. Web API 模块。

#### 4.4 前端展示端

前端用于展示定位结果，可以显示：

1. 房间平面图。
2. 参考节点位置。
3. 待定位设备当前位置。
4. 设备到各参考节点的估计距离。
5. 历史轨迹。
6. 信号强度变化曲线。

### 5. 通信协议设计

#### 5.1 蓝牙广播数据格式

BLE 广播包可以携带设备标识和少量业务数据。例如：

```text
UUID: 设备所属系统标识
Major: 区域编号
Minor: 设备编号
Tx Power: 1 米处参考发射功率
```

参考节点扫描到广播包后，不需要与设备建立连接即可获取 RSSI。广播模式的优点是功耗低、支持多个参考节点同时接收；缺点是不能保证每个广播包都被收到。

#### 5.2 参考节点上传数据格式

参考节点通过 IP 网络向服务器上传观测数据。可以使用 HTTP 或 MQTT。

HTTP JSON 示例：

```json
{
  "anchor_id": "A01",
  "anchor_x": 0.0,
  "anchor_y": 0.0,
  "device_id": "tag-001",
  "rssi": -63,
  "tx_power": -59,
  "timestamp": "2026-06-06T20:00:00+08:00"
}
其中，`anchor_id` 用于标识上传数据的参考节点，`device_id` 用于标识待定位设备，`rssi` 是测距计算的核心输入，`timestamp` 用于在服务器端进行时间窗口聚合。
```

如果使用 HTTP，优点是实现简单、便于调试；如果使用 MQTT，优点是适合多个节点持续上传小数据包，支持发布订阅模型，更适合实时物联网系统。

#### 5.3 网络传输可靠性

参考节点到服务器之间可使用 TCP，以保证观测数据可靠到达。如果系统对实时性要求高，也可以使用 UDP，但需要在应用层增加序号、时间戳和丢包处理。

从计算机网络角度看，推荐使用：

1. 蓝牙广播负责短距离无线发现。
2. TCP/IP 网络负责参考节点到服务器的数据上传。
3. HTTP 或 MQTT 负责应用层数据交换。

### 6. 测距原理设计

#### 6.1 基于 RSSI 的距离估计

RSSI 是接收信号强度。一般情况下，距离越远，RSSI 越小。可以使用路径损耗模型估计距离：

```text
d = 10 ^ ((TxPower - RSSI) / (10 * n))
```

图27 RSSI 与距离关系示意图

其中：

| 符号 | 含义 |
|---|---|
| d | 估计距离 |
| TxPower | 1 米处接收信号强度 |
| RSSI | 当前接收信号强度 |
| n | 路径损耗因子 |

在空旷环境中，n 可能接近 2；在室内环境中，由于墙壁、人体、家具、多径反射等影响，n 通常需要通过实验标定。

#### 6.2 RSSI 滤波

RSSI 会受到干扰和多径效应影响，单次测量不稳定。因此需要滤波处理：

1. 滑动平均滤波：取最近 N 次 RSSI 的平均值。
2. 中值滤波：去除突发异常值。
3. 加权平均滤波：新的测量值权重大，旧值权重小。
4. 卡尔曼滤波：适合连续移动目标的位置估计。

例如，参考节点 A 对设备 tag-001 最近 5 次 RSSI 为：

```text
-61, -63, -62, -80, -64
```

其中 `-80` 可能是瞬时干扰导致的异常值，中值滤波可以减小该异常值影响。

#### 6.3 蓝牙 5.1 到达角 AoA

如果硬件支持蓝牙 5.1 AoA，可以通过天线阵列估计信号到达角。AoA 方案比单纯 RSSI 更准确，但硬件复杂度更高，需要多天线阵列和专门算法。

在本设计中，基础方案采用 RSSI 测距；扩展方案可加入 AoA 提升精度。

### 7. 定位算法设计

#### 7.1 三边定位

如果至少有三个参考节点能够估计到目标设备距离，就可以使用三边定位。假设三个参考节点坐标为：

```text
A(x1, y1), B(x2, y2), C(x3, y3)
```

目标设备坐标为：

```text
P(x, y)
```

到三个参考节点的距离分别为：

```text
d1, d2, d3
```

则满足：

```text
(x - x1)^2 + (y - y1)^2 = d1^2
(x - x2)^2 + (y - y2)^2 = d2^2
(x - x3)^2 + (y - y3)^2 = d3^2
```

实际环境中 RSSI 测距存在误差，三个圆通常不会精确相交，因此应使用最小二乘法求近似位置。

图28 三边定位原理示意图

#### 7.2 多参考节点加权定位

当参考节点数量超过三个时，可以对多个距离估计结果加权。通常信号越强、距离越近，该参考节点测量值越可靠，权重越高。

加权思想如下：

```text
距离近的参考节点权重大
RSSI 波动小的参考节点权重大
历史误差小的参考节点权重大
```

这样可以减小单个参考节点异常测量对最终结果的影响。

#### 7.3 指纹定位

RSSI 在室内环境中受墙体、门窗、人体遮挡影响较大，单纯路径损耗模型往往不够准确。因此可以采用指纹定位。

指纹定位分两个阶段：

1. 离线采集阶段：在房间中选取多个采样点，记录每个点接收到各参考节点的 RSSI，形成指纹库。
2. 在线定位阶段：实时采集当前 RSSI 向量，与指纹库进行匹配，选择最相似的位置作为估计位置。

例如：

```text
位置 P1: A=-50, B=-70, C=-68, D=-80
位置 P2: A=-65, B=-55, C=-73, D=-77
当前位置: A=-52, B=-69, C=-67, D=-82
```

当前位置更接近 P1，因此估计设备位于 P1 附近。

指纹定位比单纯 RSSI 距离模型更适合复杂室内环境，但需要提前采集数据，维护成本更高。

### 8. 数据处理流程

服务器端数据处理流程如下：

```text
接收参考节点上传数据
-> 校验字段是否合法
-> 按 device_id 分组
-> 按时间窗口聚合同一设备数据
-> 对每个参考节点 RSSI 进行滤波
-> 将 RSSI 转换为距离
-> 使用三边定位或指纹匹配计算位置
-> 平滑位置结果
-> 保存数据库
-> 推送给前端展示
```

时间窗口可以设置为 1 秒或 2 秒。例如服务器每 1 秒计算一次设备位置，使用最近 1 秒内多个参考节点的 RSSI 数据。

### 10. 误差来源与改进方法

蓝牙定位误差主要来自以下方面：

1. 多径效应：无线信号经过墙壁、地面、家具反射后到达接收端。
2. 人体遮挡：人体会吸收 2.4GHz 信号。
3. 发射功率不稳定：不同设备蓝牙芯片和天线性能不同。
4. 环境变化：门窗开关、人员移动、设备摆放都会改变 RSSI。
5. 信道干扰：Wi-Fi、蓝牙耳机、鼠标键盘等设备也使用 2.4GHz 频段。
6. 时间同步问题：多个参考节点采集时间不一致会影响移动目标定位。

图29 室内蓝牙定位误差来源示意图

改进方法包括：

1. 增加参考节点数量。
2. 对 RSSI 进行中值滤波和滑动平均。
3. 对不同房间分别标定路径损耗因子。
4. 使用指纹定位替代单纯距离模型。
5. 使用蓝牙 5.1 AoA 提升角度测量能力。
6. 采用时间戳和序号处理乱序、延迟和重复数据。
7. 对定位结果进行轨迹平滑，避免位置点频繁跳动。

### 11. 安全性设计

蓝牙定位系统涉及设备身份和位置信息，因此需要考虑安全性。因此，系统在设计时应将蓝牙链路层地址与应用层用户身份解耦，服务器只保存应用层临时 ID 与定位结果，避免直接暴露真实设备 MAC 地址。

安全措施包括：

1. 设备 ID 不直接使用真实 MAC 地址，避免长期跟踪隐私。
2. 参考节点上传数据时使用 HTTPS 或 MQTT over TLS。
3. 服务器对参考节点进行身份认证。
4. 上传数据中增加时间戳，避免重放攻击。
5. 前端访问需要用户认证。
6. 数据库中对历史轨迹设置访问权限。
7. 对异常 RSSI 数据进行检测，防止伪造蓝牙广播干扰定位。

从计算机网络角度看，该系统不仅是无线测距问题，也是一个完整的网络应用系统，需要考虑链路层设备识别、网络层传输、应用层协议和数据安全。

### 12. 系统可行性分析

基于 BLE RSSI 的定位系统成本较低、部署方便，适合实验室和普通室内场景。其优点是硬件便宜、功耗低、支持手机和 Beacon 等常见设备；缺点是精度受环境影响较大，单纯 RSSI 测距通常只能达到米级精度。

如果实验目标是演示计算机网络和无线通信原理，RSSI 方案已经足够。如果目标是实现较高精度定位，可以进一步采用 AoA、UWB 或 Wi-Fi RTT 等技术。

### 13. 第二部分总结

蓝牙定位系统可以看作一个由无线链路层感知和 TCP/IP 网络传输组成的综合系统。蓝牙设备通过广播帧暴露设备标识，参考节点在物理层测量 RSSI，在数据链路层识别设备，在应用层形成观测数据，再通过 IP 网络上传到服务器。服务器结合滤波、测距、三边定位或指纹匹配算法计算位置，最后通过 Web 前端展示。

该方案体现了计算机网络中分层设计、地址标识、无线信道、数据封装、可靠传输、应用层协议和安全认证等知识。虽然 RSSI 测距存在误差，但通过多参考节点、滤波、标定和指纹库等方法，可以构建一个较为合理的室内蓝牙定位系统。
